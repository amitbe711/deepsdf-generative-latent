Final Project — Deep Learning for 3D Computer Vision

Generative Latent Models for DeepSDF:
Gaussian vs. Diffusion in the Small-Data Regime

This page is a from-scratch, part-by-part walkthrough of the entire codebase: every file, every method, and the reasoning behind each design choice. It is written so you can trace a single latent code all the way from "a directory of chairs" to "a trained generator that can hallucinate new chairs," and understand why each piece exists, not just what it does.

Author: Amit Benbenishti Stage 1: DeepSDF auto-decoder Stage 2: Gaussian vs. DDPM Grid: N ∈ {10,50,150} × D ∈ {16,32}

The one-sentence idea

DeepSDF can represent a shape as a latent vector z that a neural network turns into a 3D surface — but it has no idea which vectors z are "valid shapes" and which are garbage, because it never learns a distribution over z. This project bolts a second, much smaller model on top of a frozen DeepSDF decoder whose only job is to learn that distribution, so you can sample a fresh z and get a plausible new shape. Two candidate "distribution learners" (generators) are compared — a plain multivariate Gaussian (cheap, closed-form) and a DDPM (diffusion model, expensive, expressive) — across a grid of dataset sizes and latent dimensions, to find out when the expensive one is actually worth it.

Key mental model There are exactly two training phases and they never touch each other's parameters: Stage 1 learns "how to turn a code into a shape" (the decoder), Stage 2 learns "which codes are legal" (the generator). PART 1 Data pipeline — from shapes to SDF samples

DeepSDF does not train on meshes directly. It trains on a cloud of (point, signed-distance) pairs. All of Part 1 exists to produce exactly that: a tensor of 3D points and, for each one, the true signed distance to the shape's surface.

1.1 Why signed distance fields, and why a synthetic shape family

A signed distance function (SDF) represents a surface implicitly as the zero level set of a scalar field \( f:\mathbb{R}^3 \to \mathbb{R} \): \(f(\mathbf{x})\) is the distance from point \(\mathbf{x}\) to the nearest surface point, negative inside the shape and positive outside. Compared to voxels or point clouds, an SDF is resolution-free (you can query it at arbitrary precision) and differentiable, which is exactly what lets a single neural network stand in for the whole field.

The project supports two data sources, controlled by data.source in the config:

SourceWhat it isGround-truth SDFWhen to use
synthetic (default)Analytic parametric "chairs" generated in codeExact, closed-form — no mesh neededFast, fully reproducible, no preprocessing; used for the main N×D sweep
mesh_dirA directory of real meshes (e.g. a ShapeNet chairs subset)Estimated via nearest-surface-point queriesValidates the pipeline transfers to real, non-analytic geometry
Why synthetic shapes exist at all Real mesh datasets (ShapeNet) need licensing, downloading, and mesh cleanup (many meshes aren't watertight, which breaks signed-distance queries). The parametric chair family sidesteps all of that: it is a coherent shape category (so generation metrics like Coverage/MMD are meaningful — there is a real notion of "a valid chair"), it has an exact analytic SDF (no sampling error), and it can generate hundreds of variants deterministically from a seed. The exact same downstream code (Stage 1, Stage 2, metrics) is oblivious to which source produced the data.

1.2 The parametric chair family src/data/synthetic.py

Every chair is a union of axis-aligned boxes: one seat slab, one backrest, and four legs. A box SDF is computed exactly with the classic formula (used everywhere in signed-distance-field graphics): 

def _box_sdf(points, center, half):
    q = np.abs(points - center) - half
    outside = np.linalg.norm(np.maximum(q, 0.0), axis=-1)
    inside = np.minimum(np.max(q, axis=-1), 0.0)
    return outside + inside

Intuitively: q measures, per axis, how far the point sticks out past the box's half-extent. If the point is outside on any axis, outside is the Euclidean distance of that overshoot. If the point is fully inside, outside is zero and inside becomes the (negative) distance to the nearest face — giving a proper negative-inside SDF.

A whole chair is the union of several boxes, and the SDF of a union of shapes is the pointwise minimum of their individual SDFs (a point is "inside the union" if it's inside any primitive): 

class AnalyticShape:
    def sdf(self, points):
        dists = [_box_sdf(points, c, h) for c, h in self.boxes]
        return np.min(np.stack(dists, axis=0), axis=0)

ParametricChair builds the 6 boxes (seat, backrest, 4 legs) from a ChairParams dataclass (seat width/depth/height/thickness, backrest height, leg thickness). make_synthetic_collection(num_shapes, seed) draws these parameters uniformly at random from a reasonable range and returns a reproducible list of chairs — the entire "dataset" is just a seed plus this code.

When a mesh is actually needed (for surface sampling or ground-truth comparison), to_mesh() calls the same Marching-Cubes utility used everywhere else in the project (Part 4.1) on the analytic sdf() function.

1.3 Normalizing meshes into the unit sphere src/data/normalize.py

DeepSDF assumes every shape's surface lives inside the unit sphere, with a small safety margin so that near-surface point perturbations never leave the [-1,1]³ cube used at inference. normalize_mesh_to_unit_sphere does this in two steps:

  1. Translate the mesh so its bounding-box center sits at the origin.
  2. Compute the farthest vertex from the origin (radius) and uniformly scale so that radius maps to 1 / buffer (buffer ≈ 1.03, matching DeepSDF's convention), leaving ~3% headroom.

This only applies to the mesh_dir path — analytic chairs are already built inside the unit cube by construction.

1.4 SDF sampling strategy src/data/sdf_sampling.py

A network trained on points sampled uniformly at random inside the cube would waste almost all of its capacity learning the "far field", where the SDF is smooth and uninteresting. DeepSDF's Sec. 3 sampling recipe (implemented here) concentrates samples where the surface actually is:

90% of points
near the surface — sample a surface point, then perturb it with Gaussian noise
→ half of those
perturbed with a small σ = 0.005 (fine detail near the surface)
→ other half
perturbed with a larger σ = 0.02 (coarser structure slightly further out)
10% of points
drawn uniformly in [-1,1]³ — anchors the far field so the SDF doesn't blow up away from the surface

For real meshes, the true signed distance at arbitrary query points is computed via trimesh's proximity query, with the sign flipped to match the DeepSDF convention (negative inside):

def _signed_distance(mesh, points):
    query = trimesh.proximity.ProximityQuery(mesh)
    sd = query.signed_distance(points)  # trimesh: positive inside
    return -np.asarray(sd, dtype=np.float32)             # flip to DeepSDF convention

For analytic chairs, the SDF is evaluated directly with the closed-form shape.sdf(points) — no proximity query needed, no sampling error.

1.5 ShapeSDFDataset — turning shapes into training batches src/data/dataset.py

build_shape_collection(cfg, num_shapes) is the single entry point that dispatches on data.source and returns a list of dicts {"points", "sdf", "surface", "mesh"}, one per shape. ShapeSDFDataset then stacks these into contiguous tensors:

self.points

self.sdf

Training needs mini-batches that mix points from several different shapes at once (so gradients update several latent codes per step). random_batch(num_shapes, points_per_shape) does this by (1) picking a random subset of shape indices, (2) for each, picking random point indices from that shape's pre-sampled pool, and (3) concatenating everything into three flat tensors: shape_idx (which code to look up), points, and sdf targets. This is the tensor triplet that flows directly into the Stage-1 training loop.

PART 2 Stage 1 — the DeepSDF auto-decoder

Stage 1's job: learn one shared decoder network f_θ(z, x) and, simultaneously, one latent code z_i per training shape, such that f_θ(z_i, x) ≈ the true SDF of shape i at point x, for every shape in the collection.

2.1 Auto-decoder, not encoder — the key conceptual twist

A normal autoencoder has an encoder that maps a shape to a code. DeepSDF has no encoder at all. Instead, each shape is assigned a free-floating trainable vector (an embedding row, exactly like a word-embedding table), initialized from a tiny Gaussian, and that vector is optimized by gradient descent, jointly with the decoder weights, purely to minimize the reconstruction loss. This is the "auto-decoder" framework from the DeepSDF paper (Park et al., 2019).

Why this matters for Stage 2 Because there's no encoder, there is also no natural way to know what a "new" valid code looks like — an encoder would implicitly define a manifold of encodable inputs, but an auto-decoder's codes are just N free points floating in \(\mathbb{R}^D\) that happened to land somewhere useful. That's precisely the gap Stage 2 exists to fill: learn the distribution that these N points came from, so you can draw a new one.

2.2 The latent codes module src/models/latent_codes.py

LatentCodes is a thin wrapper around nn.Embedding(num_shapes, latent_dim): row i is shape i's code. It's initialized from \(\mathcal{N}(0, 0.01^2)\) — deliberately tiny, so training starts from "every shape looks almost identical" and differentiates gradually, which is more stable than starting from large random codes. 

class LatentCodes(nn.Module):
    def __init__(self, num_shapes, latent_dim, init_std=0.01):
        super().__init__()
        self.embedding = nn.Embedding(num_shapes, latent_dim)
        nn.init.normal_(self.embedding.weight, mean=0.0, std=init_std)

    def forward(self, shape_idx):
        return self.embedding(shape_idx)   # lookup: index -> code vector

Because it's an nn.Module with registered parameters, codes.parameters() plugs straight into the same Adam optimizer as the decoder — that's what makes the "joint" optimization trivial to implement.

2.3 The decoder architecture src/models/decoder.py

DeepSDFDecoder is an MLP that takes the concatenation [z ; xyz] (a (D+3)-dim vector) and predicts one scalar: the SDF value at that point, for that shape.

Depth / width
8 hidden layers × 512 units (base config), or 128 units in the smoke config
Skip connection
at layer 4, the original input [z;xyz] is concatenated back in — prevents the input signal from being "washed out" in deep layers
Weight normalization
applied to every hidden layer (not the output layer) — reparameterizes each weight as magnitude × direction, which empirically stabilizes SDF-MLP training
Activation
ReLU + dropout (p=0.2 in the base config, since DeepSDF over-fits on tiny N otherwise)
Output
a final tanh squashes the SDF prediction into (-1, 1), matching the clamped loss range
def forward(self, latent, xyz):
    inputs = torch.cat([latent, xyz], dim=-1)   # [B, D+3]
    x = inputs
    for layer, linear in enumerate(self.layers):
        if layer in self.skip_in:
            x = torch.cat([x, inputs], dim=-1)   # re-inject input at layer 4
        x = linear(x)
        if layer < self.num_layers - 1:
            x = self.activation(x)
            if self.dropout_prob > 0:
                x = self.dropout(x)
    if self.use_tanh:
        x = torch.tanh(x)
    return x

Note this network is shared across every shape in the collection — the only thing that changes per shape is which z gets concatenated in. This is what lets a single set of weights represent an entire category of shapes, and it's exactly why freezing \(\theta\) after Stage 1 and treating z as the only "sample-able" variable makes sense: the decoder is a fixed function, the codes are the data.

2.4 Geometric initialization — starting the network as a sphere

SDF-MLPs are notoriously prone to a degenerate failure mode: at random initialization, the network's output can be roughly constant everywhere in space. If the constant is never exactly zero, there is no zero level-set — Marching Cubes finds nothing, and gradients from the reconstruction loss are weak because the field doesn't respond to x in a useful way. Geometric initialization (Atzmon & Lipman 2020 "SAL"; Gropp et al. 2020 "IGR") sidesteps this by choosing the initial weights so the network starts out computing (approximately) \(f(\mathbf{x}) \approx \lVert \mathbf{x} \rVert - r\) — the exact SDF of a sphere of radius r — regardless of the latent code. 

def _geometric_init(self, linears, radius):
    last = self.num_layers - 1
    for layer, lin in enumerate(linears):
        out_dim, in_dim = lin.weight.shape
        if layer == last:
            # output layer ~ ||x|| - radius
            nn.init.normal_(lin.weight, mean=math.sqrt(math.pi) / math.sqrt(in_dim), std=1e-5)
            nn.init.constant_(lin.bias, -radius)
        else:
            nn.init.constant_(lin.bias, 0.0)
            nn.init.normal_(lin.weight, 0.0, math.sqrt(2.0) / math.sqrt(out_dim))
            if layer == 0:
                lin.weight.data[:, :self.latent_dim] = 0.0   # kill z's influence...
            if layer in self.skip_in:
                lin.weight.data[:, -self.input_dim:] = 0.0   # ...at every re-injection point

Three ideas combine here:

  • Output layer weights/bias are set so the final linear+tanh layer approximates \(\lVert h \rVert - r\) for whatever features h arrive, which is the signature of a sphere SDF when the earlier layers behave roughly like an orthogonal map.
  • Hidden layers use a specific variance (\(\sqrt{2/\text{out\_dim}}\)) derived so that, combined with ReLU, the expected norm of activations is preserved layer to layer (a geometric-init analogue of Kaiming initialization).
  • The latent columns are explicitly zeroed at layer 0 (and again wherever the skip connection re-injects the input). This is the crucial trick for an auto-decoder specifically: at step 0, every shape's code is a different tiny random vector, but zeroing out z's contribution to the first layer means the network initially ignores z entirely and produces the exact same sphere for every shape. Training then differentiates the sphere into different shapes as the codes move and the latent columns stop being zero.
Observed failure without it The project's own report notes that without geometric init, the decoder was observed to collapse to a spatially-constant field for some (N, D) cells — confirming this isn't cosmetic, it's load-bearing for training stability across the whole grid.

2.5 Loss function and training loop src/train/stage1_train.py

The per-step loss has two terms:

\[ \mathcal{L} = \underbrace{\frac{1}{|B|}\sum_{(i,\mathbf{x})\in B} \bigl|\,\mathrm{clamp}(f_\theta(\mathbf{z}_i,\mathbf{x}),\delta) - \mathrm{clamp}(s(\mathbf{x}),\delta)\,\bigr|}_{\text{clamped-}L_1\text{ reconstruction}} \;+\; \underbrace{\lambda \cdot \frac{1}{|B|}\sum_i \lVert \mathbf{z}_i \rVert_2^2}_{\text{code regularization}} \] 
def clamped_l1_loss(pred, target, delta):
    pred_c = torch.clamp(pred, -delta, delta)
    target_c = torch.clamp(target, -delta, delta)
    return torch.abs(pred_c - target_c).mean()
TermPurpose
Clamping to δ = 0.1Far from the surface, exact SDF magnitude doesn't matter — only its sign / rough distance. Clamping stops the loss from being dominated by large far-field errors and focuses capacity near the surface where precision matters.
L1 instead of L2More robust to the sampling noise inherent in near-surface perturbation.
Code regularization λ‖z‖²Pulls every code toward the origin. This is exactly a Gaussian-prior MAP estimate: minimizing ‖z‖² is equivalent to maximizing log N(z; 0, I). This is also precisely why a plain Gaussian is a principled Stage-2 candidate, not an arbitrary baseline — it's the same prior implicitly assumed by this regularizer.

The training loop itself is a standard joint-optimization Adam loop, with separate learning rates for decoder and codes (both 1e-3 by default):

  1. Sample a batch: shapes_per_batch random shapes × points_per_shape random points each, via dataset.random_batch(...).
  2. Look up each point's shape code: z = codes(idx).
  3. Forward through the decoder: pred = decoder(z, pts).
  4. Compute clamped-L1 reconstruction loss + code regularization, sum them.
  5. Backprop through both the decoder parameters and the embedding table, and step one shared Adam optimizer (with per-group learning rates).
  6. Every log_every steps, record {step, loss, recon, reg} for later plotting.

The function returns {"decoder", "codes", "history"} — everything Stage 2 and the evaluation code need.

PART 3 Stage 2 — two generators over the frozen latent space

At this point Stage 1 has produced a frozen decoder f_θ and a fixed matrix of N codes {z_i} ⊂ ℝ^D. Stage 2 never touches the decoder or the SDF data again — it treats {z_i} as an ordinary N × D point cloud in latent space and fits a generative model to it.

3.1 Why not just sample \(\mathcal{N}(0, I)\) directly?

The code regularizer during Stage 1 nudges codes toward a standard normal, so in principle \(z \sim \mathcal{N}(0, I)\) should be a reasonable prior already. In practice the learned codes rarely fill an isotropic, axis-aligned Gaussian — they can be anisotropic, correlated, shifted off-center, or (with small N) clustered in a way a fixed identity covariance can't capture. Sampling blindly from \(\mathcal{N}(0,I)\) then frequently lands outside the region the decoder was actually trained on, producing broken or empty shapes. Both Stage-2 generators exist to fix this by fitting to the actual empirical distribution of {z_i} instead of assuming it.

3.2 Generator A — the Gaussian prior src/models/gaussian_prior.py

This is the "minimal" generator: fit a multivariate Gaussian \(\mathcal{N}(\mu, \Sigma)\) to {z_i} by maximum likelihood (i.e. sample mean and sample covariance), then sample from it in closed form. Two covariance modes are supported:

ModeFittingSampling
diagonalper-dimension mean and standard deviationz = μ + ε ⊙ σ, ε ~ N(0,I)
full (default)full D×D covariance matrix + a small ridge reg·I for numerical stability, then Cholesky-factorized: Σ = L Lᵀz = μ + Lε
def fit(self, codes):
    codes = codes.detach().float()
    self.mean = codes.mean(dim=0)
    centered = codes - self.mean
    cov = (centered.t() @ centered) / max(codes.shape[0] - 1, 1)
    cov = cov + self.reg * torch.eye(dim)          # ridge: keeps Σ positive-definite
    self.chol = torch.linalg.cholesky(cov)          # Σ = L Lᵀ

@torch.no_grad()
def sample(self, num_samples, generator=None):
    eps = torch.randn(num_samples, dim, generator=generator)
    return self.mean + eps @ self.chol.t()          # z = μ + Lε

The ridge term reg · I matters at small N: with fewer samples than dimensions (or close to it), the raw sample covariance can be singular or ill-conditioned, and Cholesky factorization would fail. Adding a small multiple of the identity guarantees positive-definiteness at negligible cost to fidelity.

Why this is a meaningful baseline, not a strawman Capturing full linear correlations between latent dimensions is already more than the isotropic N(0,I) the code regularizer assumes — it's the best possible unimodal, linear-Gaussian fit to the codes. If a DDPM can't beat this by much, it means the empirical code distribution doesn't have enough non-Gaussian structure (multi-modality, curvature, heavy tails) for the diffusion model's extra expressiveness to pay off — which is exactly the scientific question this project asks.

3.3 Generator B — the latent DDPM src/models/ddpm.py

This reuses the exact denoising-diffusion formulation from Ho et al. (2020) (the same construction behind HW3's image diffusion), applied to plain D-dimensional vectors instead of images. There is no image, no U-Net, no convolutions — just an MLP that denoises points in latent space.

Forward process — turning a clean code into noise

A noise schedule \(\{\beta_t\}_{t=1}^{T}\) defines \(\alpha_t = 1-\beta_t\) and \(\bar\alpha_t = \prod_{s\le t}\alpha_s\). The forward ("noising") process has a closed form that lets you jump straight to any timestep without simulating intermediate steps:

\[ q(\mathbf{z}_t \mid \mathbf{z}_0) = \mathcal{N}\!\left(\sqrt{\bar\alpha_t}\,\mathbf{z}_0,\; (1-\bar\alpha_t) I \right) \quad\Longrightarrow\quad \mathbf{z}_t = \sqrt{\bar\alpha_t}\,\mathbf{z}_0 + \sqrt{1-\bar\alpha_t}\,\boldsymbol{\epsilon}, \;\; \boldsymbol{\epsilon}\sim\mathcal{N}(0,I) \] 
def q_sample(self, z_0, t, noise):
    sqrt_ac = self.sqrt_alphas_cumprod[t].unsqueeze(-1)
    sqrt_om = self.sqrt_one_minus_alphas_cumprod[t].unsqueeze(-1)
    return sqrt_ac * z_0 + sqrt_om * noise

Noise schedule — cosine vs. linear

Two schedules are implemented; the default is the cosine schedule (Nichol & Dhariwal, 2021), which keeps \(\bar\alpha_t\) from collapsing to zero too quickly at early timesteps — this was found empirically to give more stable training than the original linear schedule, especially with very few diffusion steps (100–200 here, vs. thousands typical for images): 

if schedule == "cosine":
    steps = num_timesteps + 1
    x = torch.linspace(0, num_timesteps, steps)
    s = 0.008
    f = torch.cos(((x / num_timesteps) + s) / (1 + s) * math.pi / 2) ** 2
    alphas_cumprod = f / f[0]
    betas = 1 - (alphas_cumprod[1:] / alphas_cumprod[:-1])
    return betas.clamp(1e-5, 0.999)

The denoiser network — MLPDenoiser

A small residual MLP predicts the noise \(\boldsymbol{\epsilon}_\phi(\mathbf{z}_t, t)\) that was added at timestep t. The timestep is encoded with a sinusoidal embedding (the same trick as Transformer positional encodings — a fixed, non-learned mapping from a scalar integer to a smooth, high-dimensional vector so the network can distinguish neighboring timesteps easily): 

def sinusoidal_time_embedding(timesteps, dim):
    half = dim // 2
    freqs = torch.exp(-math.log(10000.0) * torch.arange(half) / max(half - 1, 1))
    args = timesteps.float()[:, None] * freqs[None, :]
    return torch.cat([torch.cos(args), torch.sin(args)], dim=-1)

class MLPDenoiser(nn.Module):
    def forward(self, z_t, t):
        temb = self.time_mlp(sinusoidal_time_embedding(t, self.time_embed_dim))
        h = self.input_proj(z_t)
        for block in self.blocks:
            h = h + self.act(block(h + temb))   # residual block, time injected additively
        return self.output_proj(h)

Each residual block adds the time embedding into the hidden state before a linear+SiLU, then adds the result back as a residual — this is the standard "FiLM-lite" way small diffusion MLPs condition on time without full attention or normalization layers.

Training objective

The DDPM training loss is beautifully simple — predict the noise that was added, regress with MSE:

\[ \mathcal{L}_{\text{DDPM}} = \mathbb{E}_{\mathbf{z}_0,\, t\sim U(1,T),\, \boldsymbol\epsilon\sim\mathcal{N}(0,I)} \left\lVert \boldsymbol\epsilon_\phi(\mathbf{z}_t, t) - \boldsymbol\epsilon \right\rVert_2^2 \] 
def loss(self, z_0):
    t = torch.randint(0, self.num_timesteps, (z_0.shape[0],))
    noise = torch.randn_like(z_0)
    z_t = self.q_sample(z_0, t, noise)
    noise_pred = self.model(z_t, t)
    return torch.mean((noise_pred - noise) ** 2)

Every training step: sample a batch of real codes, pick a random timestep per example, noise them, ask the network to recover the noise, backprop. train_ddpm (in src/train/stage2_ddpm.py) wraps this in a standard Adam loop, logging loss every 200 steps.

Reverse process — ancestral sampling (generation)

To generate a new code, start from pure noise \(\mathbf{z}_T \sim \mathcal{N}(0,I)\) and iteratively denoise, one timestep at a time, using the trained network's noise prediction to estimate the reverse-process mean, and adding calibrated noise back in (except at the very last step):

\[ \boldsymbol\mu_\theta(\mathbf{z}_t, t) = \frac{1}{\sqrt{\alpha_t}}\left(\mathbf{z}_t - \frac{\beta_t}{\sqrt{1-\bar\alpha_t}}\boldsymbol\epsilon_\phi(\mathbf{z}_t,t)\right), \qquad \mathbf{z}_{t-1} = \boldsymbol\mu_\theta(\mathbf{z}_t,t) + \sqrt{\tilde\beta_t}\,\mathbf{n},\;\; \mathbf{n}\sim\mathcal{N}(0,I) \] 
@torch.no_grad()
def sample(self, num_samples, device="cpu", generator=None):
    z = torch.randn(num_samples, self.latent_dim, device=device, generator=generator)
    for step in reversed(range(self.num_timesteps)):
        t = torch.full((num_samples,), step, dtype=torch.long)
        noise_pred = self.model(z, t)
        mean = (z - self.betas[step] / self.sqrt_one_minus_alphas_cumprod[step] * noise_pred) \
               / torch.sqrt(self.alphas[step])
        if step > 0:
            noise = torch.randn(num_samples, self.latent_dim, generator=generator)
            z = mean + torch.sqrt(self.posterior_variance[step]) * noise
        else:
            z = mean   # no noise on the final step
    return z

This loop runs num_timesteps (100–200) sequential network evaluations per sample — the fundamental reason diffusion is far more expensive than the Gaussian, which samples in a single matrix multiply. That cost/benefit trade-off is the entire point of the experiment grid.

PART 4 From a latent code back to a mesh

Both generators produce the same kind of object: a vector z ∈ ℝ^D. Turning that into something visual/measurable is the same two-step process regardless of which generator produced it.

4.1 Marching Cubes wrapper src/marching.py

eval_sdf_on_grid builds a dense resolution³ grid over [-bound, bound]³, and evaluates any SDF-like callable on it in memory-safe chunks (default 65536 points at a time — important on Colab GPUs with limited VRAM). sdf_grid_to_mesh then runs Marching Cubes (Lorensen & Cline, 1987) — the classic algorithm that walks every cube (voxel) in the grid and, based on which corners are positive/negative, emits triangles approximating the zero level-set — at level=0 to get a watertight-ish triangle mesh, converting grid coordinates back to world coordinates. It gracefully returns None if the whole volume is on one side of zero (no surface exists, e.g. the decoder produced garbage).

4.2 sample.py — the full z → mesh / point-cloud pipeline

decode_sdf_grid is the decoder-specific analogue of eval_sdf_on_grid: it broadcasts one latent code across every grid point and queries the network in chunks:

lat = latent.expand(pts.shape[0], -1)     # same code, every grid point
sdf = decoder(lat, pts).squeeze(-1).cpu()

decode_mesh chains this into Marching Cubes, and decode_point_cloud further samples a surface point cloud from the resulting mesh — the representation the metrics in Part 5 actually consume. This single utility is reused by: reconstruction evaluation (decoding known codes), generation evaluation (decoding sampled codes from either generator), and the qualitative gallery figure.

PART 5 Measuring quality — the metrics suite

Two different questions are being asked, and each gets its own metric family.

5.1 Reconstruction metrics src/metrics/reconstruction.py

"Given a shape's own trained code, how well does the decoder reproduce that exact shape?" This isolates Stage-1 quality, independent of either generator.

Chamfer distance
symmetric nearest-neighbor distance between two point clouds — for every point in cloud A, distance to its nearest neighbor in B, averaged, plus the same in reverse. Computed efficiently with a scipy.spatial.cKDTree.
Volumetric IoU
rasterize both meshes' occupancy on a shared grid (via mesh.contains(grid)) and compute intersection-over-union of the occupied voxels.
\[ \mathrm{CD}(A,B) = \frac{1}{|A|}\sum_{a\in A}\min_{b\in B}\lVert a-b\rVert^2 + \frac{1}{|B|}\sum_{b\in B}\min_{a\in A}\lVert a-b\rVert^2 \]

5.2 Generation metrics (Achlioptas et al., 2018) src/metrics/generation.py

"Given a batch of freshly sampled shapes, do they look like they came from the same distribution as real shapes?" This is what actually differentiates the Gaussian from the DDPM. All three metrics are built on a pairwise Chamfer-distance matrix between a set of generated point clouds and a held-out reference set of real shapes.

MetricDefinitionIdeal valueWhat it captures
MMD (Minimum Matching Distance)for each reference shape, distance to its closest generated shape, averaged0 (lower is better)fidelity — are generated shapes close to real ones?
Coveragefraction of reference shapes that are the nearest neighbor of some generated shape1.0 (higher is better)diversity / mode coverage — does generation reproduce the full variety, or collapse to a few modes?
1-NN accuracypool generated+reference clouds together; for each, check whether its nearest neighbor has the same label (generated vs. real); average accuracy of this classifier0.5 (indistinguishable)a genuine two-sample test — 1.0 or 0.0 both mean the two distributions are trivially separable (bad); 0.5 means real and fake are statistically indistinguishable under Chamfer distance

The reference set is built from a disjoint random seed (cfg.seed + 10000 in run_grid.py) so it is never contaminated by the training shapes.

valid_ratio, a fourth silent metric Not from Achlioptas et al., but tracked throughout: the fraction of sampled codes that actually decoded to a non-empty mesh at all (some samples, especially poorly-fit or out-of-distribution ones, can decode to "no zero crossing found"). A generator that samples garbage codes will have a low valid_ratio even before its shapes are compared to references.

6.2 run_grid.py — step by step

For every combination of (N, D) in the grid, run_cell() performs, in order:

  1. Seed everything deterministically from cfg.seed (so every cell starts from a controlled, comparable RNG state).
  2. Build the training data: build_shape_collection(cfg, N) → ShapeSDFDataset.
  3. Train Stage 1: train_stage1(cfg, dataset, D) → decoder + codes + loss history.
  4. Evaluate reconstruction: decode up to eval.max_recon_shapes of the training codes back to meshes, compare Chamfer + IoU against ground truth.
  5. Build a held-out reference set (disjoint seed) for generation metrics.
  6. Fit the Gaussian on the frozen codes; sample eval.num_generated new codes; decode each to a point cloud; compute Coverage / MMD / 1-NN / valid_ratio against the reference set.
  7. Train the DDPM on the same frozen codes; sample the same number of new codes; same generation evaluation.
  8. Persist everything: a resumable checkpoint.pt (decoder weights, code embedding, config, N, D) and a metrics.json (all metrics + both loss histories) per cell, under outputs/<run>/N{N}_D{D}/.

After all cells finish, a single summary.json aggregates every cell's results — this is the one file make_figures.py actually needs to read.

CLI flags --only-N / --only-D let you re-run a single cell in isolation (useful for debugging one problematic combination without re-running the whole grid).

PART 7 Engineering scaffolding

7.1 Reproducibility & checkpoints src/utils/seed.py, src/utils/io.py

seed_everything(seed) seeds Python's random, NumPy, and both CPU and CUDA Torch RNGs in one call — used at the start of every grid cell so results are as reproducible as GPU nondeterminism allows.

save_checkpoint / load_checkpoint are thin wrappers around torch.save/torch.load, but by convention every checkpoint bundles everything needed to reconstruct the run: decoder state dict, code embedding state dict, the full config dict, and latent_dim/num_shapes — so a checkpoint from a Colab run that got disconnected mid-grid can be reloaded and re-evaluated without re-training, and figures can be regenerated long after training finished.

7.2 prepare_data.py — the optional real-mesh utility

Not part of the main pipeline — a standalone helper for working with a real mesh directory (e.g. ShapeNet chairs): --inspect-dir reports how many files in a directory actually load as valid, non-empty triangle meshes (many real-world mesh files fail this silently), and --cache pre-computes and stores SDF samples as a compressed .npz so repeated experiments over the same real dataset don't re-run the (slower) proximity-query sampling every time.

7.3 Authorship accounting src/thirdparty/README.md

This folder is intentionally empty — its README exists purely as a paper trail: it states explicitly that every algorithmic file under src/ (decoder, codes, both generators, sampling, marching cubes, metrics, grid runner) is original implementation, lists the external library dependencies used as tools rather than vendored code (PyTorch, NumPy, SciPy, scikit-image, trimesh, Matplotlib, PyYAML), and lists the papers whose algorithms were re-implemented from scratch rather than copied (DeepSDF, DDPM, the cosine schedule, the Achlioptas generation-metric protocol).

Glossary

SDF
Signed Distance Function — scalar field whose zero level-set is a surface; negative inside, positive outside.
Auto-decoder
A generative setup with no encoder — per-sample latent codes are free parameters, optimized jointly with the decoder via gradient descent.
Latent code (z)
A learned vector in ℝ^D that, fed into the shared decoder alongside a query point, identifies which shape to reconstruct.
Marching Cubes
Algorithm that extracts a triangle mesh from a scalar field sampled on a grid, at a chosen level (here, 0).
Geometric init
A weight initialization scheme that makes an SDF-MLP start out as an exact sphere, avoiding the "constant field, no surface" failure mode.
Clamped L1 loss
L1 loss on SDF values after clamping both prediction and target to ±δ, focusing training near the surface.
DDPM
Denoising Diffusion Probabilistic Model — learns to reverse a fixed noising process by predicting the noise added at each timestep.
Cosine schedule
A choice of noise-variance schedule β_t that decays ᾱ_t smoothly, improving training stability vs. a linear schedule.
Chamfer distance
Symmetric nearest-neighbor distance between two point clouds; base metric for both reconstruction and generation evaluation.
Coverage / MMD / 1-NN
Distribution-level generation metrics comparing a set of generated shapes to a held-out reference set.
valid_ratio
Fraction of sampled latent codes that decode to a non-empty mesh at all.
